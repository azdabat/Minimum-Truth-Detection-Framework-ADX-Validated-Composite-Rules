T1003.001
LSASS Memory — OS Credential Dumping
CREDENTIAL ACCESS · SUBSTRATE-FIRST DETECTION
▼
🔵 CORE BASELINE COMPOSITE
Non-System Process OpenProcess to LSASS with VM_READ Access Anchor
📎 INSERT RULE LINK · KQL · MDE DeviceEvents 🔗 MITRE T1003.001 🔗 Mimikatz (reference)
🔵 BLUE TEAM
LSASS holds plaintext credentials, NTLM hashes, and Kerberos tickets in memory. Any non-system process opening a handle to lsass.exe with PROCESS_VM_READ access is structurally anomalous. MDE surfaces this as a DeviceEvents OpenProcessApiCall event.

Composite anchor: non-whitelisted process + PROCESS_VM_READ handle to lsass. Reinforcement: SeDebugPrivilege enabled on the accessing process, handle duplication to lsass, minidump creation via MiniDumpWriteDump, or the accessing process was itself injected.
🔴 RED TEAM
Why LSASS: A single dump on a domain controller or admin workstation yields NTLM hashes for every user authenticated to that machine. Pass-the-hash with a domain admin hash = complete domain compromise in minutes.

Modern evasion techniques: Nanodump uses direct syscalls to avoid API hooks. comsvcs.dll MiniDump (LOLBin — no Mimikatz binary needed): rundll32 C:\Windows\System32\comsvcs.dll MiniDump [lsass_pid] C:\Temp\lsass.dmp full. SilentProcessExit registry technique triggers WER dump without touching LSASS directly.
Mimikatz sekurlsa Nanodump pypykatzcomsvcs MiniDump APT28 Sandworm

DCSync — Rogue DC Replication Request
T1003.006
▼
📎 INSERT RULE LINK 🔗 T1003.006 DCSync
⚡ KILL CHAIN: Credential Access (post domain admin — never touches DC disk)
🔵 BLUE TEAM
DCSync abuses the AD replication protocol to request credential data from a DC without touching LSASS directly. Minimum truth: a non-DC machine or non-DC account making MS-DRSR replication calls — specifically GetNCChanges for sensitive attributes. Monitor Event 4662 with replication permission GUIDs (1131f6aa-9c07-11d1-f79f-00c04fc2dcd2 — Replicating Directory Changes All) from non-DC source accounts.
🔴 RED TEAM
Why DCSync over LSASS dump: Never touches DC disk or memory directly. mimikatz lsadump::dcsync /user:krbtgt requests the krbtgt hash via legitimate DC replication protocol. With krbtgt hash → Golden Ticket → persistent domain access for up to 10 years.

Required privileges: Domain admin, or an account with Replicating Directory Changes All — sometimes misconfigured on service accounts.

SAM / LSA Secrets Registry Dump
T1003.002 · T1003.004
▼
📎 INSERT RULE LINK 🔗 T1003.002 SAM
⚡ KILL CHAIN: Credential Access (local account hashes — no domain admin needed)
🔵 BLUE TEAM
SAM database and LSA Secrets are accessed via registry hive export. Detection: reg save HKLM\SAM, reg save HKLM\SYSTEM, reg save HKLM\SECURITY commands from non-administrative tools — these three registry hives together allow offline extraction of all local account hashes and service account credentials (LSA Secrets). Also monitor Volume Shadow Copy access to these hive files.
🔴 RED TEAM
Why SAM/LSA over LSASS: Requires only local admin (not SeDebugPrivilege). The local Administrator hash from SAM is often identical across all machines in an environment if built from the same image — one local admin hash gives lateral movement to every machine before LAPS is deployed.

LSA Secrets contains service account credentials for all services configured to run under domain accounts — often highly privileged accounts with weak passwords.
Impacket secretsdump local Mimikatz lsadump::sam CrackMapExec --sam

NTDS.dit Extraction — Domain Credential Database
T1003.003
▼
📎 INSERT RULE LINK 🔗 T1003.003 NTDS
⚡ KILL CHAIN: Credential Access (complete domain hash dump — all users)
🔵 BLUE TEAM
NTDS.dit is the Active Directory database containing hashes for every domain account. The file is locked by the OS while running — extraction requires VSS (Volume Shadow Copy) access or ntdsutil. Detection: VSS snapshot creation on a DC followed by access to the shadow copy path containing NTDS.dit, or ntdsutil snapshot commands from non-backup service accounts. One successful NTDS.dit dump = all domain account hashes offline for cracking.
🔴 RED TEAM
The holy grail: NTDS.dit contains every domain user's NTLM hash. Offline cracking with Hashcat + wordlists + rules on a GPU cracks weak passwords in minutes. Even uncracked hashes can be used for PtH attacks.

ntdsutil "ac i ntds" "ifm" "create full c:\temp\dump" q q — creates a full AD snapshot including NTDS.dit and SYSTEM hive for offline extraction.
Impacket secretsdump ntdsutil IFM VSS shadow copy extraction APT29
T1558.003 · T1558.004
Kerberoasting / AS-REP Roasting — Ticket-Based Hash Extraction
CREDENTIAL ACCESS · STEAL OR FORGE KERBEROS TICKETS
▼
🔵 CORE BASELINE COMPOSITE
Anomalous TGS-REQ Volume — RC4 Encrypted Service Tickets Anchor
📎 INSERT RULE LINK · KQL · Sentinel (Event ID 4769) 🔗 T1558.003 Kerberoasting 🔗 T1558.004 AS-REP Roasting
🔵 BLUE TEAM
Any authenticated domain user can request Kerberos service tickets (TGS) for any SPN — encrypted with the service account's NTLM hash for offline cracking. Detection: anomalous volume of 4769 (TGS requests) for RC4-encrypted tickets (encryption type 0x17) from a single source within a short time window. RC4 is the crack-friendly encryption — legitimate modern services use AES256 (0x12). A user requesting RC4 tickets for multiple service accounts is almost certainly Kerberoasting.
🔴 RED TEAM
Why powerful: Requires zero elevated privileges — any domain user can Kerberoast. Service accounts frequently have weak passwords and excessive AD rights (Domain Admin "for convenience").

Rubeus Kerberoast: Rubeus.exe kerberoast /rc4opsec /outfile:hashes.txt — requests RC4 tickets for all SPNs, outputs Hashcat-compatible format for offline cracking. A weak service account password cracked in minutes.
Rubeus kerberoast Impacket GetUserSPNs PowerView Get-DomainSPNTicket APT41

AS-REP Roasting — Pre-Auth Disabled Accounts
T1558.004
▼
📎 INSERT RULE LINK 🔗 T1558.004
⚡ KILL CHAIN: Credential Access (no authentication required — unauthenticated attack)
🔵 BLUE TEAM
AS-REP Roasting targets accounts with Kerberos pre-authentication disabled (DONT_REQ_PREAUTH flag). The KDC returns an AS-REP encrypted with the user's password hash — without verifying the requestor's identity first. Detection: AS-REQ requests (Event 4768) without pre-authentication data for accounts, especially from external IP addresses or unexpected source hosts. Also audit AD for accounts with pre-authentication disabled — they should be zero in most environments.
🔴 RED TEAM
No credentials needed: Unlike Kerberoasting, AS-REP Roasting requires no existing domain credentials — it can be performed from outside the domain. Any account with pre-authentication disabled returns a crackable hash to an unauthenticated attacker. Rubeus.exe asreproast /format:hashcat identifies and extracts all vulnerable accounts.
Rubeus asreproast Impacket GetNPUsers

Password Spraying — Low-and-Slow Auth Attack
T1110.003
▼
📎 INSERT RULE LINK 🔗 T1110.003 Password Spraying
⚡ KILL CHAIN: Initial Access / Credential Access (bypass lockout threshold)
🔵 BLUE TEAM
Password spraying tries one password against many accounts — staying below lockout thresholds. Detection: single source IP/host generating 4625 (failed logon) events against multiple different accounts within a time window. The spray pattern is orthogonal to brute force: one password × many users vs many passwords × one user. Composite anchors on: >10 unique account failures from a single source in 30 minutes with <3 attempts per account.
🔴 RED TEAM
Standard spray passwords: Season+Year! (Summer2024!), CompanyName1!, Welcome1, Password1!. These meet complexity requirements and are set by users who just want to satisfy the password policy.

O365/Entra spray: MSOLSpray targets Microsoft Online authentication — bypasses on-prem detection entirely, generates no on-prem event logs.

T1555.003 · T1539
Browser Credential & Cookie Theft
CREDENTIAL ACCESS · CREDENTIALS FROM PASSWORD STORES
▼
🔵 CORE BASELINE COMPOSITE
Browser Profile Database Access by Non-Browser Process Anchor
📎 INSERT RULE LINK · KQL · MDE 🔗 T1555.003 🔗 T1539 Steal Web Session Cookie
🔵 BLUE TEAM
Browser credential databases (Chrome's Login Data SQLite file, Firefox's logins.json) are targeted for stored credentials and session cookies. Minimum truth: a non-browser process accessing Chrome/Firefox/Edge profile directories and reading credential or cookie database files. Reinforcement: process then initiates network connections (exfiltrating the stolen data), or process copies the database to TEMP before reading (bypassing file locks).
🔴 RED TEAM
Why browser credentials: Users store hundreds of credentials in browsers — VPNs, cloud consoles, internal applications, personal banking. Session cookie theft enables authentication bypass without needing credentials at all — bypasses MFA.

Cookie theft for MFA bypass: Steal authenticated session cookies from browser profile → replay in attacker's browser → authenticated as victim. Increasingly used for O365/Azure access bypassing conditional access policies.
