Impact composites — particularly VSS deletion and mass file rename — should be wired to automated SOAR isolation responses. The cost of a false positive (a legitimate backup tool triggering isolation) is dramatically lower than the cost of a missed ransomware encryption event. 🔵 BLUE TEAM — Last intervention window
VSS (Volume Shadow Copy) deletion is the canonical pre-encryption signal — ransomware deletes shadow copies first to prevent recovery without paying. Minimum truth composite: VSS deletion command (vssadmin delete shadows /all /quiet, wmic shadowcopy delete, bcdedit /set recoveryenabled no) — trigger immediate SOAR isolation.

Encryption in-progress detection: file rename velocity >100 files/minute with extension change to unknown extension (.locked, .encrypted, random 6-char extension) — this is encryption happening right now. Combined composite: VSS deletion + mass rename = automated isolation trigger, no analyst approval required.

Pre-encryption signals (earlier window): stopping backup services (net stop "Backup Exec", net stop VSS), stopping database services (SQL, Exchange) to release file locks for encryption, disabling Windows Recovery Environment.
🔴 RED TEAM — Full ransomware deployment sequence
Standard Ransomware-as-a-Service pre-encryption sequence:
1. Verify domain admin, confirm target hosts accessible via CrackMapExec
2. Exfiltrate data (double extortion — already complete by this stage)
3. Deploy EDR killer (AuKill/BYOVD) across domain via GPO
4. Deploy ransomware binary simultaneously to all hosts
5. Each host executes: vssadmin delete shadows /all /quiet
6. Stop backup/AV/database services
7. Begin encryption (AES-256 per-file with RSA-encrypted session key)
8. Drop ransom note in every encrypted directory

Time from VSS deletion to full domain encryption: 45 minutes (LockBit 3.0 — optimised multi-threaded encryption) to 4 hours (older families). The VSS deletion event is the last reliable intervention point.

Destructive Wiper / MBR Overwrite
T1561 · T1561.002
▼
📎 INSERT RULE LINK 🔗 T1561.002 Disk Structure Wipe
⚡ KILL CHAIN: Impact (nation-state / sabotage — no ransom, pure destruction)
🔵 BLUE TEAM
Destructive wipers overwrite the MBR (Master Boot Record) or file system structures to render systems unbootable — or overwrite file contents with zeros/random data. Unlike ransomware, there is no decryption key. Detection: raw disk write operations to the first sectors of physical disks (MBR at LBA 0), IOCTL codes targeting disk devices from user-space processes, format.com or diskpart executing with destructive parameters from non-administrative contexts. Process writing zeros or random data to high volumes of files (without extension change pattern of ransomware).
🔴 RED TEAM
Nation-state sabotage tool: Used by Sandworm (Russia) against Ukrainian infrastructure — NotPetya (2017, disguised as ransomware but purely destructive), WhisperGate (2022, Ukrainian government targets), HermeticWiper (2022, pre-invasion). No ransom payment possible — pure destruction is the intent. Also used in destructive attacks by Iranian actors against Saudi Aramco (Shamoon wiper, 2012 and 2016).

Service Disruption / DDoS Amplification
T1489 · T1498
▼
📎 INSERT RULE LINK 🔗 T1489 Service Stop
⚡ KILL CHAIN: Impact (disrupt operations — ransomware pressure tactic or sabotage)
🔵 BLUE TEAM
Service disruption via bulk service termination is a ransomware pressure tactic and sabotage technique. Detection: bulk net stop commands across multiple critical services in rapid succession (backup services, database services, security services, production web services), sc.exe stopping services with start= disabled (prevents restart), taskkill targeting specific process lists. A batch of service stops across >5 critical services in 60 seconds is a high-confidence impact indicator.
🔴 RED TEAM
Ransomware service-stop list: Standard ransomware includes hardcoded lists of 50-100+ services to terminate — SQL Server, Exchange, Oracle, backup agents, AV services, VSS — to unlock files for encryption and prevent backup restoration. The service stop batch executes from a single command script in seconds, immediately before encryption begins.

Boot Recovery Disable — bcdedit / Recovery Env
T1490
▼
📎 INSERT RULE LINK 🔗 T1490 Inhibit System Recovery
⚡ KILL CHAIN: Impact (pre-encryption recovery prevention)
🔵 BLUE TEAM
bcdedit /set {default} recoveryenabled No and bcdedit /set {default} bootstatuspolicy ignoreallfailures prevent Windows from automatically entering recovery mode after failures — meaning a ransomware-encrypted system cannot boot to recovery environment. Detection: bcdedit.exe executing with recovery-disabling parameters from a non-IT-management process context. This is an almost universally malicious operation in enterprise environments — legitimate IT uses Group Policy for boot configuration, not ad-hoc bcdedit commands.
🔴 RED TEAM
Executed as part of the standard pre-encryption preparation script alongside VSS deletion. Prevents victims from booting to Windows Recovery Environment to attempt file recovery or system restore. Combined with VSS deletion and backup service termination, creates a recovery-hostile environment that maximises pressure to pay the ransom.
