BYOVD — Bring Your Own Vulnerable Driver
DEFENSE EVASION · PRIVILEGE ESCALATION · SUBSTRATE-FIRST · MTDF NOVEL RESEARCH
▼
🔵 CORE BASELINE COMPOSITE
BYOVD Driver Load + EDR Kill Chain Composite — Kernel Callback Deregistration Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE · LOLDRIVER Composite 🔗 MITRE T1068 🔗 T1562.001 🔗 LOLDrivers Project 📄 SilverFox/ValleyRAT R&D Publication 📄 See Novel Research: SilverFox BYOVD
🔵 BLUE TEAM — Why BYOVD requires a dedicated composite
BYOVD is the apex defense evasion technique: load a legitimate but vulnerable signed driver to gain kernel-level code execution, then use that ring-0 access to kill EDR processes or disable the kernel callbacks that EDR relies on for visibility.

Minimum truth composite: signed driver loaded from non-system path + kernel callback deregistration attempt OR driver service creation by non-system process + immediate termination of security product process within 5 minutes.

The 30-day Atomic Sentinel Layer is critical: a driver dropped and staged is caught even if execution is delayed by days — defeating the BYOVD staging pattern used by SilverFox/ValleyRAT.

LOLDRIVER composite: cross-references loaded driver against MISP supply chain feed of known vulnerable driver hashes AND certificate thumbprints — immune to byte-flip hash mutation (see cousin rule).
🔴 RED TEAM — Why BYOVD is the attacker's EDR solution
EDR products run protected processes at kernel level — user-space tools cannot terminate them. BYOVD bypasses this by exploiting a vulnerability in a signed, trusted driver to gain ring-0 (kernel) execution. From the kernel, everything is accessible: terminate any process, disable callbacks, read any memory, operate invisibly.

Full SilverFox/ValleyRAT kill chain (MTDF original research):
1. Signed loader drops vulnerable driver (e.g. gdrv.sys, iqvw64e.sys, zemana.sys)
2. Loader exploits driver vulnerability → kernel shellcode execution
3. Kernel shellcode deregisters EDR callbacks (PsSetCreateProcessNotifyRoutine, ObRegisterCallbacks)
4. EDR is completely blind — all subsequent payloads execute undetected
5. BYOVD driver unloaded to reduce footprint
6. ValleyRAT RAT deployed with full persistence

EDR killer tools sold as-a-service: Spyboy Terminator, AuKill, Terminator — purchased on criminal forums, deployed by ransomware operators with no kernel programming required.

Byte-Flip Driver Mutation — Hash Evasion
T1027 · T1068
▼
📎 INSERT RULE LINK · LOLDRIVER Composite 📄 Novel Research: SilverFox Byte-Flip
⚡ KILL CHAIN: Defense Evasion (hash-based IOC bypass) → Privilege Escalation
🔵 BLUE TEAM
Attackers mutate known-vulnerable drivers by flipping bytes in non-critical sections — the hash changes, functionality preserved, all hash-based IOC blocks defeated. The MTDF LOLDRIVER composite shifts to behaviour-based substrate detection: driver signing certificate thumbprint (the vulnerable driver's cert remains valid even after byte mutation), driver import table analysis for known vulnerable API patterns, and kernel callback deregistration behaviour post-load. This is part of the original MTDF research — standard vendor detection relies entirely on hash lists which byte-flipping trivially defeats.
🔴 RED TEAM
The problem defenders have: LOLDrivers project and MITRE publish hash lists of known vulnerable drivers. Blocking by hash is trivially bypassed: change a single byte in a padding section of the driver binary — the exploit still works, the hash is now unique and unblocked by every IOC-based control.

Why MTDF catches it anyway: Certificate thumbprint detection. The driver's code-signing certificate cannot be changed without re-signing — and the vulnerable certificate is what identifies the driver family. Behaviour-based: any driver making DeviceIoControl calls that result in kernel callback deregistration is flagged regardless of hash.

LOLDRIVER Master Composite — MISP Supply Chain Integration

📎 INSERT RULE LINK · LOLDRIVER KQL Master Rule 🔗 LOLDrivers.io 📄 MTDF MISP Supply Chain Lab
⚡ KILL CHAIN: Defense Evasion → Privilege Escalation (all BYOVD variants)
🔵 BLUE TEAM
The LOLDRIVER master composite is a tiered rule consolidating multiple detection layers: (1) hash-based matching against MISP supply chain feed of known vulnerable drivers, (2) certificate thumbprint matching defeating byte-flip evasion, (3) driver load path anomaly (non-system directory), (4) driver service creation by non-system process, (5) post-load kernel behaviour. The MISP integration ensures the IOC feed updates automatically — new vulnerable drivers discovered by the community are immediately reflected in detection without manual rule updates.
🔴 RED TEAM
Why supply chain integration matters: New vulnerable drivers are discovered continuously — LOLDrivers.io lists hundreds. Without automated feed integration, the defender is always reactive. The MTDF MISP lab closes this gap: threat intelligence production feeds directly into composite rule generation, creating a live defensive posture rather than a static ruleset.

All variants achieve the same outcome: malicious code executing inside a trusted process. The substrate-first minimum truth — OpenProcess + memory write + remote thread/APC queue — is structurally similar across all variants despite different API paths.
🔵 CORE BASELINE COMPOSITE
Cross-Process Memory Write + Remote Thread Creation Substrate Anchor
📎 INSERT RULE LINK · KQL · MDE DeviceEvents 🔗 MITRE T1055
🔵 BLUE TEAM
Process injection is the substrate-first detection problem: malicious code executing inside another process has no visible command line. The minimum truth is the API call sequence: OpenProcess → VirtualAllocEx → WriteProcessMemory → CreateRemoteThread. MDE DeviceEvents surfaces these as individual telemetry events — the composite joins them on source/target PID within a 30-second window.

High-fidelity reinforcement: target process is a system process (lsass.exe, svchost.exe, explorer.exe), source process has no legitimate cross-process reason to interact with target, or source process was itself spawned from an anomalous parent (the injection chain starts from the original execution event).
🔴 RED TEAM
Why inject? Code running inside explorer.exe or svchost.exe appears to come from a trusted, signed process. Network connections from explorer look normal. AV/EDR that analyses process behaviour may not inspect the injected thread separately.

OSCP-level chain (Cobalt Strike):
1. Initial beacon running as standard user
2. inject [PID] x64 [listener] → Cobalt Strike allocates memory in explorer.exe, writes shellcode, creates remote thread
3. Second beacon now runs inside explorer.exe — all subsequent C2 traffic attributed to trusted process
4. steal_token [PID] — impersonate higher-privileged user token from memory

Modern variants use direct syscalls (SysWhispers2) to bypass EDR API hooks — calling kernel functions directly without going through ntdll.dll where EDR hooks are placed.

Process Hollowing — Legitimate Process Shell
T1055.012
▼
📎 INSERT RULE LINK 🔗 T1055.012
⚡ KILL CHAIN: Defense Evasion → Execution (legitimate process shell)
🔵 BLUE TEAM
Process hollowing creates a legitimate process in suspended state, unmaps its memory, and replaces it with malicious code before resuming. Minimum truth: CreateProcess with CREATE_SUSPENDED flag → NtUnmapViewOfSection → VirtualAllocEx → WriteProcessMemory → ResumeThread. The resulting process has a legitimate name and path but malicious executable in memory — PID and path match a trusted binary while the memory image does not.
🔴 RED TEAM
Why hollowing over standard injection: The process name, path, and parent all match a legitimate Windows binary (e.g. svchost.exe). Task Manager shows a real svchost. The malicious code is only visible via memory forensics — the on-disk binary is legitimate, only the runtime memory image is replaced.

Used heavily by banking trojans (Dridex, TrickBot) and sophisticated RATs. The hollowed process can be a child of a legitimate parent, making the process tree appear entirely normal.

APC Injection / Early Bird
T1055.004
▼
📎 INSERT RULE LINK 🔗 T1055.004
⚡ KILL CHAIN: Defense Evasion (API hook bypass variant)
🔵 BLUE TEAM
APC (Asynchronous Procedure Call) injection queues shellcode to execute in the context of a thread in another process when that thread enters an alertable state. Early Bird variant: APC queued before the process's main thread executes — runs before most EDR hooks are established. Minimum truth: NtQueueApcThread or QueueUserAPC calls from an unexpected source process to a target process thread.
🔴 RED TEAM
Why APC/Early Bird: Bypasses EDR user-mode hooks because the shellcode executes before EDR's hook initialisation completes (Early Bird variant). Does not use CreateRemoteThread — evades detection rules that specifically look for that API.

Used by APT34 and Cobalt Strike's spawnto technique.

PPID Spoofing — Parent Process Masquerade
T1134.004
▼
📎 INSERT RULE LINK 🔗 T1134.004
⚡ KILL CHAIN: Defense Evasion (process tree manipulation)
🔵 BLUE TEAM
PPID spoofing creates a child process with a falsified parent PID — making cmd.exe spawned by malware appear as a child of explorer.exe in the process tree. Detection: process tree inconsistency — the stated PPID no longer exists, the process creation time is earlier than the stated parent's creation time, or the process session ID doesn't match the parent's session ID. MDE's process tree can surface these inconsistencies.
🔴 RED TEAM
Defeats detection rules based on parent-child process relationships (e.g. "Word spawning PowerShell"). By spoofing explorer.exe as the parent of the malicious process, the detection rule sees explorer → powershell which may be whitelisted, rather than winword → powershell which is highly suspicious.

Tool: SelectMyParent, Cobalt Strike's ppid setting in malleable profiles.

Obfuscation & Deobfuscation/Decode
DEFENSE EVASION · OBFUSCATED FILES OR INFORMATION
▼
🔵 CORE BASELINE COMPOSITE
High-Entropy Payload Execution + Decode Operation Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1027 🔗 T1140 Deobfuscate/Decode
🔵 BLUE TEAM
Obfuscation detection targets the deobfuscation action, not the obfuscated content. The minimum truth: a process performing a decode operation (base64, XOR, RC4) on a blob that is then immediately written to disk or executed in memory. Script Block Logging captures PowerShell deobfuscation at runtime — the deobfuscated content is logged even if the script itself was encoded. Reinforce with: output entropy analysis, decode-to-execute timing (<5 seconds), decode operation in TEMP directory.
🔴 RED TEAM
Obfuscation layers serve two purposes: bypassing static signature detection AND obscuring intent from log inspection. Multiple layers are stacked: base64 → XOR → string substitution → character concatenation. Invoke-Obfuscation generates thousands of functionally equivalent but syntactically unique PowerShell variants — each has a unique hash, none match signatures.

Steganographic variant (MTDF original research): payload encoded in LSB of image file pixels — see Novel Research: Stego Loader Ecosystem.

Indicator Removal — Log Tampering & Evidence Destruction
DEFENSE EVASION · INDICATOR REMOVAL ON HOST
▼
🔵 CORE BASELINE COMPOSITE
Windows Event Log Clear / Modification Anchor
📎 INSERT RULE LINK · KQL · Sentinel (Event ID 1102 / 104) 🔗 T1070.001 Clear Windows Event Logs
🔵 BLUE TEAM
Log clearing events are high-fidelity signals with very few legitimate explanations outside scheduled maintenance windows. Event ID 1102 (Security log cleared) and Event ID 104 (System log cleared) should alert immediately. The minimum truth composite also covers: wevtutil.exe clearing specific logs (wevtutil cl Security), PowerShell Clear-EventLog cmdlet, and direct log file deletion/modification.

Cousin coverage: attackers who cannot clear logs will attempt to flood them with noise (log flooding as cousin) or disable auditing entirely (T1562.002) — both covered in this ecosystem.
🔴 RED TEAM
Kill chain position: Post-exploitation cleanup, typically after lateral movement objectives are complete and before the attacker prepares to exit or when they realise they may have been detected.

Why attackers clear logs: Remove evidence of lateral movement (4624 logon events), privilege escalation (4672 special privileges), process execution (4688), and LSASS access (4656). A clean Security log forces investigators to rely on EDR telemetry and network logs instead.

Ransomware standard: Clearing logs is part of most ransomware operator playbooks — executed alongside VSS deletion before encryption to hamper incident response.

📎 INSERT RULE LINK 🔗 T1562.002
⚡ KILL CHAIN: Defense Evasion (prospective — before malicious action)
🔵 BLUE TEAM
Disabling audit policy prevents events from being generated — subtler than clearing logs because there is no evidence of the change unless audit policy modification itself is audited (Event ID 4719). Monitor for auditpol.exe /set /category:* /success:disable /failure:disable, Group Policy modifications to audit settings, and registry changes to HKLM\SYSTEM\CurrentControlSet\Services\EventLog.
🔴 RED TEAM
Prospective vs retrospective evasion: Log clearing is retrospective (removes existing evidence). Audit disable is prospective — stop generating evidence for future actions. Preferred by patient APT actors who plan to remain in the environment for extended periods and need sustained reduced visibility rather than periodic cleanup.

Timestomping — File Metadata Manipulation
T1070.006
▼
📎 INSERT RULE LINK 🔗 T1070.006
⚡ KILL CHAIN: Defense Evasion (forensic anti-analysis)
🔵 BLUE TEAM
Timestomping modifies file MACE (Modified/Accessed/Created/Entry) timestamps to blend malicious files into the legitimate file landscape or confuse incident response timeline reconstruction. Detection: NTFS $STANDARD_INFORMATION and $FILE_NAME timestamp discrepancy — these are separate NTFS attributes and timestomping tools typically only modify $STANDARD_INFORMATION, leaving $FILE_NAME intact. MDE surfaces timestamp anomalies in file creation events.
🔴 RED TEAM
Used primarily by APT actors who plan for long-term access and forensic investigation resistance. Files backdated to match system file creation dates blend into the file landscape — a dropped DLL with a timestamp of 2019 appears less suspicious than one created at 3am today.

Tool: Metasploit timestomp, Cobalt Strike's file attribute modification, custom PowerShell using [System.IO.File]::SetCreationTime().

Masquerading — Blend Into Legitimate Processes
DEFENSE EVASION · MASQUERADING · NAME / PATH SPOOFING
▼
🔵 CORE BASELINE COMPOSITE
Process Name / Path Mismatch Anchor — Typosquat and Path Anomaly
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1036
🔵 BLUE TEAM
Masquerading exploits analyst trust in familiar process names. Minimum truth: a process named like a known system binary but executing from a non-standard path (e.g. svchost.exe running from C:\Users\Public\), OR a process name that is a typosquat of a system binary (scvhost.exe, lsass_.exe, svch0st.exe). The composite cross-references process name against expected execution paths from a whitelist of known-good binaries.

Reinforcement: process lacks the digital signature of the binary it is impersonating, the process icon/version information does not match, or the process is spawning from an unexpected parent.
🔴 RED TEAM
Why masquerade: Analysts monitoring process lists for anomalies will scan past familiar names. A malicious binary named svchost.exe in the wrong directory will only be caught by path verification — which many analyst workflows skip.

Common masquerade targets: svchost.exe, lsass.exe, csrss.exe, explorer.exe, winlogon.exe — all expected to run as SYSTEM, all expected to have multiple instances, all trusted enough to be mentally skipped by tired analysts.

Also used for C2 malware naming: MicrosoftEdgeUpdate.exe, GoogleUpdateHelper.exe in scheduled tasks or startup entries.
