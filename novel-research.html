NOVEL RESEARCH #1
SilverFox / ValleyRAT — BYOVD EDR Kill Chain
▼
📎 SilverFox-BYOVD-Core-Hunt.kql 📎 SilverFox-BYOVD-Advanced-Hunt.kql 📎 SilverFox_ValleyRAT_Stager.kql 📎 SilverFox_ValleyRAT_ScriptStager.kql 📎 SilverFox_ValleyRAT_AdvancedRule.kql 📎 SilverFox-BYOVD-Vs-Polymorphic-Malware README 📄 View Full Repository on GitHub
Threat Overview
Actor: Chinese-nexus threat group targeting finance, cryptocurrency, and enterprise Windows environments in targeted campaigns.

Classification: Multi-stage RAT with BYOVD kernel-level EDR kill chain. SilverFox is the loader — ValleyRAT is the final payload.

Why it matters: Seven distinct evasion layers operating in sequence. Each layer individually is detectable — together they form a chain designed to defeat defence-in-depth at every stage. Signature-based controls fail entirely.

🔴 FULL KILL CHAIN — Attacker Perspective
1
Initial Delivery: Phishing delivers a signed loader binary (legitimate code-signing certificate, custom packer). Binary appears as a legitimate installer or update package. Hash is unique — not in any IOC database.
2
DLL Sideloading: Signed loader drops a malicious DLL alongside a legitimate Microsoft-signed binary. When that binary executes, it loads the malicious DLL via Windows DLL search order. The executing process is signed and trusted — EDR sees a legitimate application loading a DLL.
3
Vulnerable Driver Drop: Sideloaded DLL drops a known-vulnerable signed kernel driver (e.g. gdrv.sys — Gigabyte driver, iqvw64e.sys — Intel Network Adapter). Driver is Microsoft-signed (or signed by a trusted hardware vendor). Byte-flipping applied to the binary before drop — hash is now unique, defeating all hash-based IOC blocks. Certificate thumbprint remains unchanged.
4
BYOVD Exploitation: Driver vulnerability exploited to gain kernel-mode code execution (ring-0). From the kernel, attacker executes shellcode that calls PsSetCreateProcessNotifyRoutine with a NULL pointer to deregister EDR process-creation callbacks, and ObRegisterCallbacks deregistration to remove EDR object access callbacks. EDR is now completely blind — it cannot see new process creation, memory reads, or network connections.
5
Driver Unload: BYOVD driver unloaded immediately after callback deregistration to reduce forensic footprint. The EDR remains blind even after the driver is removed — the kernel callbacks were already deregistered.
6
ValleyRAT Deployment: With EDR neutralised, ValleyRAT RAT deployed with full persistence (registry run key, scheduled task, or WMI subscription). RAT establishes HTTPS C2 with custom JA3 profile mimicking legitimate browser. Beacon interval 2-4 hours with 30% jitter — evades statistical beaconing detection.
7
Long-Term Persistence: ValleyRAT provides modular plugin capability — keylogging, screen capture, credential theft, lateral movement. Actor conducts intelligence collection over weeks or months. No obvious destructive activity — designed for long-term persistent access.
🔵 DETECTION COVERAGE — 7 Rules, 3 Validation Tiers
TIER 1 — STAGER DETECTION
SilverFox_ValleyRAT_Stager.kql + SilverFox_ValleyRAT_ScriptStager.kql
Signed loader anomaly composite: legitimate code-signing certificate on a binary executing from a user-writable path, combined with parent process anomaly (browser, email client) and DLL sideload detection. The sideloaded DLL is loaded by a legitimate signed application — detection anchors on the DLL appearing in a non-standard path alongside the legitimate binary, combined with subsequent network or child process activity.
TIER 2 — BYOVD CHAIN DETECTION
SilverFox-BYOVD-Core-Hunt.kql + SilverFox-BYOVD-Advanced-Hunt.kql
Core rule: vulnerable driver service creation from a non-system process → driver file written to non-system path → driver loaded → kernel callback deregistration attempt (DeviceIoControl to driver handle with known exploit IOCTL codes).

Advanced rule: certificate thumbprint matching — the vulnerable driver's code-signing certificate thumbprint is matched regardless of file hash mutation. Byte-flip evasion defeated: the certificate thumbprint cannot be changed without re-signing, and the known-vulnerable certificate is the identifying artefact.

30-day Atomic Sentinel Layer: driver drop event tracked independently — a driver staged and not immediately loaded is caught when it is eventually executed, even days later. Defeats the staged BYOVD pattern.
TIER 3 — VALLEYRAT RAT DETECTION
SilverFox_ValleyRAT_AdvancedRule.kql
ValleyRAT post-EDR-kill behaviour: process injection into a system process (PPID spoofed to appear as legitimate parent), C2 beaconing with statistical periodicity analysis (2-4 hour intervals, jitter-tolerant composite), persistent connection to Ethereum or custom C2 infrastructure. Module loading behaviour — ValleyRAT's plugin architecture loads additional DLLs into the host process on demand.
SPECIAL RESEARCH — BYOVD VS POLYMORPHIC MALWARE
SilverFox-BYOVD-Vs-Polymorphic-Malware README + LOLDRIVER Master Composite
Original research publication documenting why all signature-based controls fail against SilverFox/ValleyRAT: (1) signed loader — trusted cert, (2) LOLBin sideload — known-good application, (3) Microsoft-signed vulnerable driver, (4) byte-flip hash mutation — unique hash, (5) encrypted C2 — no payload inspection possible, (6) kernel-blind EDR — no telemetry generated. The LOLDRIVER master composite (MISP-integrated) provides the behaviour-based substrate detection that defeats all seven evasion layers.

NOVEL RESEARCH #2
EtherRAT / React2Shell — Blockchain C2 via CVE-2025-55182
▼
📎 React2Shell_CVE-2025-55182.kql 📎 React2Shell_CVE-2025-55182_AnalyticMode.kql 📎 React2Shell_CVE-2025-55182_HuntMode.kql 📎 React2Shell_CVE-2025-55182_L3_Hunt_V.kql 📎 React2Shell + EtherRAT Combined Detection 📎 Test Handbook.md 📎 Threat Modelling Attacks.md 📄 Full Repository: React2Shell_EtherRAT
⚡ WHY THIS MATTERS — EtherRAT delivers C2 via the Ethereum blockchain. There is no C2 server to sinkhole, no domain to block, no IP to blacklist. The C2 infrastructure IS the public blockchain. This represents a fundamental shift in C2 architecture that defeats all traditional C2 detection approaches based on domain/IP reputation, domain age, or sinkholing.
CVE-2025-55182 — REACT2SHELL IIS EXPLOIT
Vulnerability: React2Shell IIS exploit chain allowing remote code execution on IIS web servers via malformed React component rendering request.

Impact: Unauthenticated RCE on IIS → SYSTEM-level code execution → EtherRAT deployment → persistent blockchain C2.

Attack surface: Any IIS server running the vulnerable React rendering component, including enterprise intranet applications, SharePoint installations, and custom web applications.
ETHERRAT — BLOCKCHAIN C2 ARCHITECTURE
C2 mechanism: Commands embedded in Ethereum transaction input data field. EtherRAT polls a public blockchain API (Infura, Alchemy) at regular intervals for transactions addressed to its designated wallet.

Why uncensorable: The Ethereum blockchain is a distributed, censorship-resistant public ledger. No single entity can remove a transaction or block the attacker's address. The C2 channel cannot be taken down — it can only be detected.
🔴 FULL KILL CHAIN — Why Blockchain C2 Is a Step Change
1
Exploitation: Attacker sends malformed React rendering request to target IIS server → CVE-2025-55182 triggers → w3wp.exe (IIS worker process) executes attacker shellcode → SYSTEM-level code execution achieved without authentication.
2
EtherRAT Deployment: Shellcode downloads and executes EtherRAT from a staging server (one-time connection — staging server can be taken down after delivery with no impact on ongoing C2). EtherRAT establishes on the target system and begins polling.
3
Blockchain Polling: EtherRAT polls a public Ethereum API endpoint (mainnet.infura.io or eth-mainnet.alchemyapi.io) at randomised intervals. All traffic is HTTPS to a legitimate blockchain infrastructure provider — indistinguishable from legitimate blockchain application traffic. No dedicated C2 server exists.
4
Command Delivery: Attacker sends an Ethereum transaction (cost: fractions of a cent in gas fees) to EtherRAT's designated wallet address, with the encoded command in the transaction input data field. The transaction is immediately visible on the public blockchain — anyone can read it, but only EtherRAT knows to look for it and how to decode it.
5
Command Execution: EtherRAT detects the new transaction, decodes the command from input data, executes it on the target system, encodes the output in a new transaction sent back to the attacker's monitoring address. Full bidirectional C2 entirely via public blockchain infrastructure.
6
Persistence: EtherRAT installs persistence (scheduled task or WMI subscription). Even if the initial IIS vulnerability is patched, the RAT continues operating. The C2 channel remains active indefinitely — the Ethereum blockchain cannot be taken down or the attacker's address blocked without blockchain-level governance changes.
🔵 DETECTION COVERAGE — React2Shell + EtherRAT Multi-Layer
IIS EXPLOIT DETECTION — React2Shell CVE-2025-55182
React2Shell_CVE-2025-55182.kql + L3 Hunt Variant
Exploit detection: w3wp.exe (IIS worker process) spawning cmd.exe, PowerShell, or any non-IIS child process — high-fidelity signal with very low false positive rate. Reinforcement: IIS request log entries matching exploit path pattern, anomalous HTTP response codes from IIS application pools, w3wp.exe initiating outbound network connections (staging server download).

Analytic mode vs Hunt mode: Analytic rule (scheduled, lower threshold, higher FP tolerance for coverage) and Hunt mode rule (higher threshold, for proactive threat hunting sessions) provided as separate KQL variants.
BLOCKCHAIN C2 DETECTION — EtherRAT Polling
React2Shell + EtherRAT Combined Detection
Blockchain API polling detection: Outbound HTTPS connections to known Ethereum API providers (mainnet.infura.io, eth-mainnet.alchemyapi.io, rpc.ankr.com/eth) from non-developer host processes — specifically from IIS worker processes, Windows services, or scheduled task processes. Regular interval polling (beaconing pattern detection applied to blockchain API calls).

Transaction parsing detection: Processes reading and executing content parsed from JSON responses containing Ethereum transaction input fields — the decode-to-execute pattern that is structurally identical to the PowerShell download cradle pattern but using blockchain API responses as the delivery mechanism.

Why standard C2 detection fails: No malicious domain. No suspicious IP. HTTPS to Infura.io — a legitimate, widely-used Ethereum infrastructure provider used by thousands of legitimate blockchain applications. Domain/IP reputation-based detection is completely blind.

NOVEL RESEARCH #3
Steganographic Loader Ecosystem — Image-Based Payload Delivery
▼
📎 Stego-Image-Loader.kql 📎 Stego_Image_Based_Loader.kql 📎 Stego_Image_Loader_Phase2.kql 📎 Stego_.NET_In_Memory_Hunt.kql 📎 Stego_Attack_Report.md 📎 Stego_Threat_Hunters_Incident_Response_SOP.md 📎 Web-Based Stego-Loader Explained.md 📄 Full Repository: Stego_Threat_Analysis_And_Hunter_Pack
🔵 DETECTION PRINCIPLE — Steganographic payloads cannot be detected by scanning the carrier image — it is a valid, displayable image with correct headers. Detection is entirely behavioural: the anomalous action of a scripting engine reading an image file and executing the parsed result. No signature of the image itself is possible or useful.
THE TECHNIQUE
Malicious payload encoded in the least-significant bits (LSB) of image pixel data (PNG/BMP/JPEG). The carrier image is a valid image file that displays normally — correct headers, valid colour data, viewable in any image viewer.

The payload is invisible to the human eye and to all signature-based security controls. No embedded PE, no obvious malicious strings, no detection by hash or YARA signatures that scan file content.
WHY IT EVADES
No malicious file ever exists on disk — payload only materialises in memory after LSB extraction. Network DLP inspects the image download and sees a legitimate PNG. Email gateway scans the attachment and finds a valid image. AV scans the file and finds no malicious signatures. The only detectable artifact is the extraction script's behaviour.
🔴 FULL KILL CHAIN — Multi-Phase Delivery
1
Delivery: Phishing email, compromised website, or drive-by download delivers the carrier image file (or links to a hosted image). The image displays normally if viewed — a corporate logo, a banner, an invoice header image. No suspicious file extension, no warning from email gateway.
2
Dropper Execution: A small dropper script (VBS, PowerShell, or a LOLBin) downloads or reads the image file and begins LSB extraction. The dropper may itself be delivered via a separate phishing mechanism or be embedded in a malicious Office document. The dropper reads the image pixel data using native image APIs or .NET System.Drawing.
3
LSB Extraction: Dropper reads each pixel's least-significant bit, assembles them into bytes, decodes from base64 or XOR-encoded payload. The extracted data is shellcode or a full PE (executable) — loaded directly into memory via VirtualAlloc/WriteProcessMemory. No decoded file is ever written to disk.
4
In-Memory Execution: Payload executes entirely in memory — Cobalt Strike shellcode, a RAT, or a next-stage loader. If .NET-based, loaded via Assembly.Load() from the decoded byte array — the .NET in-memory load leaves a specific telemetry footprint detectable by the Stego_.NET_In_Memory_Hunt.kql rule.
5
Persistence and C2: In-memory payload establishes persistence and C2 using standard techniques. The steganographic delivery chain has left no forensic artifacts — no malicious files on disk, no suspicious network traffic during delivery (the image download was legitimate), no signature matches.
🔵 DETECTION COVERAGE — Multi-Phase Detection Suite
PHASE 1 — IMAGE ACCESS ANOMALY
Stego-Image-Loader.kql + Stego_Image_Based_Loader.kql
Minimum truth: A scripting engine (PowerShell, VBScript, cscript.exe) or a non-image-editing process opening an image file (PNG/BMP/JPEG/GIF) and then executing or loading content derived from that file access within 30 seconds.

Reinforcement: Image file accessed from TEMP or AppData (not a user's documents folder), image file downloaded immediately before access (browser or BITS download then immediate script access), process making the access was spawned from an Office application or phishing delivery vector.

Statistical entropy analysis of the image file itself is supplementary — LSB-encoded images have slightly higher pixel value entropy in the LSB layer than natural images, detectable via post-alert forensic analysis.
PHASE 2 — IN-MEMORY EXECUTION
Stego_Image_Loader_Phase2.kql + Stego_.NET_In_Memory_Hunt.kql
Phase 2 anchor: The image-reading process or its child executing shellcode from a memory region that was populated by data read from an image file. MDE DeviceEvents surfaces VirtualAlloc + WriteProcessMemory + execute sequence from a process that previously performed image file reads.

.NET in-memory variant: Assembly.Load() called from a process that read image pixel data — the byte array origin can be traced to the image read operation in process telemetry. The .NET assembly loaded has no corresponding file on disk — the MDE "image load" event shows no file path.
INCIDENT RESPONSE
Stego_Threat_Hunters_Incident_Response_SOP.md + Stego_Attack_Report.md
Full IR procedure for confirmed stego-loader compromise: evidence collection (process memory dump of the extraction script's runtime — the decoded payload may still be in process heap), extraction of the carrier image for forensic LSB analysis, network traffic capture for C2 identification, and scope assessment (how many hosts received the same carrier image from the same delivery mechanism). Web-Based Stego-Loader Explained.md documents the web-delivery variant where the image is served from a compromised website rather than delivered via email.
