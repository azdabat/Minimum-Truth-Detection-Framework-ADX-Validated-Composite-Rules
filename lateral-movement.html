SHARED INTENT: Remote Code Execution via Authenticated Channel
🔵 DETECTION DOCTRINE
The shared intent detection anchor: an authenticated connection to a host outside normal peer-to-peer patterns, followed by process execution on the destination. The specific protocol (SMB/DCOM/WinRM) becomes secondary — the behavioural pattern of authentication + remote execution from an unexpected source is the minimum truth that cuts across all substrates.
🔴 ATTACKER SUBSTRATE PIVOT
Firewall blocks SMB port 445 → pivot to WinRM port 5985. WinRM blocked → use DCOM (random high ports). DCOM too noisy → use WMI (same auth, no service install). All blocked → pivot to RDP (3389) or VNC. A detection strategy that covers only SMB leaves 6 cousin substrates exposed.
T1021.002
SMB / Windows Admin Shares — PsExec Pattern
LATERAL MOVEMENT · REMOTE SERVICES · SMB SUBSTRATE
▼
🔵 CORE BASELINE COMPOSITE
Admin Share Access + Service Install Composite — PsExec Pattern Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 MITRE T1021.002 🔗 PsExec Documentation
🔵 BLUE TEAM
SMB lateral movement via admin shares produces a characteristic event chain: 4624 type 3 logon → 5140 share access (C$ or ADMIN$) → 7045 service installation → 4697 service run. The composite joins these four events on source IP + destination host within a 120-second window.

High-fidelity signal: a workstation (not a known management server) accessing ADMIN$ on another workstation using privileged credentials is nearly always malicious. Reinforce with: service binary written to ADMIN$ immediately before installation, service name randomised (8 random characters), service deleted within 60 seconds of execution.
🔴 RED TEAM
PsExec attack chain (OSCP standard):
1. Attacker has NTLM hash or cleartext credential
2. psexec.py DOMAIN/admin@192.168.1.50 -hashes :NTLMhash
3. Impacket drops service binary to ADMIN$, creates service, executes → SYSTEM shell
4. Service deleted after execution to reduce footprint

Modern evasion: Attackers using Impacket's smbexec avoid writing a binary to disk by executing commands directly via Service Control Manager — no file drop, only the service creation event.

DCOM Lateral Movement — MMC20/ShellWindows
T1021.003
▼
📎 INSERT RULE LINK 🔗 T1021.003 DCOM
⚡ KILL CHAIN: Lateral Movement (SMB substitute when port 445 blocked)
🔵 BLUE TEAM
DCOM lateral movement uses Windows Distributed Component Object Model to invoke remote COM objects that execute commands. Detection: mmc.exe or explorer.exe on destination host spawning unexpected child processes after an inbound DCOM activation request (authentication event 4624 type 3 correlated with DCOM activation in DCOM event log). The DCOM MMC20.Application and ShellWindows objects are most commonly abused — monitor their activation from non-standard source hosts.
🔴 RED TEAM
Why DCOM: Uses random high ports (RPC dynamic allocation) — harder to block via firewall than SMB port 445. No service installation (no event 7045). The execution appears as mmc.exe or explorer.exe spawning the command — appears legitimate.

PowerShell execution via DCOM:
$com = [System.Activator]::CreateInstance([type]::GetTypeFromProgID("MMC20.Application","192.168.1.50"))
$com.Document.ActiveView.ExecuteShellCommand("powershell",$null,"-enc [payload]","7")

WinRM Lateral Movement — PSRemoting
T1021.006
▼
📎 INSERT RULE LINK 🔗 T1021.006 WinRM
⚡ KILL CHAIN: Lateral Movement (quiet, encrypted, no service install)
🔵 BLUE TEAM
WinRM lateral movement generates: 4624 type 3 logon on destination + wsmprovhost.exe spawning child processes. The key signal is wsmprovhost.exe — the WinRM provider host — spawning cmd.exe or PowerShell on the destination. Correlate with inbound port 5985 (HTTP) or 5986 (HTTPS) connections from unexpected source hosts. In most enterprises, legitimate WinRM sources are a small known set of management servers.
🔴 RED TEAM
Why WinRM over PsExec: No service installation, no file write to ADMIN$, uses legitimate Windows Remote Management infrastructure (port 5985 looks like web traffic), encrypted by default. Cobalt Strike's jump winrm64 deploys a beacon to a remote host over WinRM in a single command using a stolen credential or hash.

Impacket: winrm.py DOMAIN/admin@192.168.1.50 -hashes :hash — interactive WinRM shell, all traffic encrypted.

RDP Lateral Movement — Hijacking & Shadowing
T1021.001
▼
📎 INSERT RULE LINK 🔗 T1021.001 RDP
⚡ KILL CHAIN: Lateral Movement (GUI access, credential-based)
🔵 BLUE TEAM
RDP lateral movement generates 4624 logon type 10 (RemoteInteractive) events. Minimum truth: RDP logon from a workstation to another workstation (expected pattern: admin workstation → server, not workstation → workstation). Also monitor for RDP session hijacking via tscon.exe — allows taking over another user's disconnected RDP session without knowing their password.
🔴 RED TEAM
RDP session hijacking (no credentials needed): With SYSTEM privileges, tscon.exe [session_id] /dest:[current_session] takes over any disconnected RDP session. A domain admin who left an RDP session disconnected can be hijacked — attacker gains their session without knowing the password, MFA, or any credential.

Ransomware operators frequently use RDP for interactive pre-deployment reconnaissance — mapping shares, identifying backup servers, manually running BloodHound.

All five use stolen credential material (hash or Kerberos ticket) to authenticate without knowing the plaintext password. The stolen material may come from any credential access technique — this ecosystem is the consumption layer of the credential access ecosystem.
🔵 CORE BASELINE COMPOSITE
Anomalous Logon Source + Admin Share / Remote Service Access Composite
📎 INSERT RULE LINK · KQL · Sentinel (Event ID 4624 type 3 + 5140) 🔗 T1550.002 PtH 🔗 T1550.003 PtT
🔵 BLUE TEAM
Pass-the-Hash produces a type 3 (network) NTLM logon using a hash rather than a password. Detection composite: 4624 type 3 NTLM logon from unexpected source host for a privileged account, immediately followed by admin share access (C$, ADMIN$, IPC$) or remote service installation (7045) within a 60-second window.

Cross-ecosystem correlation: The hash used here was likely stolen from LSASS in the credential access phase — the 30-day Atomic Sentinel Layer links the LSASS access event to this lateral movement event, providing full kill chain context in a single alert.
🔴 RED TEAM
PtH chain (Impacket, OSCP standard):
psexec.py DOMAIN/admin@192.168.1.50 -hashes :NTLMhash
Authenticates with NTLM hash, drops service binary to ADMIN$, starts it, returns SYSTEM shell. No password needed — the hash itself is the authentication material.

Pass-the-Ticket (Rubeus):
Rubeus.exe ptt /ticket:[base64_ticket]
Injects a stolen Kerberos TGT into the current logon session. All subsequent Kerberos authentications use the stolen ticket — completely transparent to the network, no NTLM traffic generated.

Golden Ticket — krbtgt Hash Exploitation
T1558.001
▼
📎 INSERT RULE LINK 🔗 T1558.001 Golden Ticket
⚡ KILL CHAIN: Persistence + Lateral Movement (domain-wide, survives password resets)
🔵 BLUE TEAM
Golden Tickets are Kerberos TGTs forged using the krbtgt account hash — valid for any service, any user, for up to 10 years. Detection signals: Kerberos tickets with anomalous lifetimes (>10 hours default max), tickets for users that don't exist in AD, tickets with mismatched domain SIDs, or tickets renewed after the krbtgt password was reset (golden ticket survives one password reset, requires two). Monitor event 4769 for service tickets requested with RC4 encryption by accounts that should use AES256.
🔴 RED TEAM
Why Golden Ticket is the ultimate persistence: With the krbtgt hash, the attacker can authenticate as any user to any service in the domain — including non-existent users with arbitrary group memberships. Persists through password resets of all user accounts (only resetting krbtgt twice invalidates existing tickets).

Obtained via: Mimikatz lsadump::dcsync /user:krbtgt (DCSync) or direct NTDS.dit dump from a Domain Controller.

Silver Ticket — Service Account Ticket Forgery
T1558.002
▼
📎 INSERT RULE LINK 🔗 T1558.002 Silver Ticket
⚡ KILL CHAIN: Lateral Movement (targeted service access, stealthier than Golden)
🔵 BLUE TEAM
Silver Tickets are forged Kerberos service tickets (TGS) for a specific service, using that service account's NTLM hash. Stealthier than Golden Tickets — never contacts the KDC (no 4769 event), the forged ticket goes directly to the service. Detection: service tickets appearing without a corresponding TGT request, service tickets for hosts that the authenticated user has no legitimate access to, or service ticket encryption anomalies.
🔴 RED TEAM
Use case: Attacker has compromised a service account hash (e.g. SQL service account via Kerberoasting). Creates a Silver Ticket for the SQL service — can authenticate as any user to that SQL server without contacting the DC or KDC. The service validates the ticket locally using its own hash — DC is never involved, no authentication logs generated on DC.

Overpass-the-Hash — NTLM to Kerberos Conversion
T1550.002
▼
📎 INSERT RULE LINK 🔗 T1550.002
⚡ KILL CHAIN: Lateral Movement (Kerberos-only environments, NTLM blocked)
🔵 BLUE TEAM
Overpass-the-Hash uses an NTLM hash to request a Kerberos TGT — converting NTLM credential material into a Kerberos ticket. Detection: a Kerberos TGT request (event 4768) immediately following an NTLM authentication, especially from a host where the user is not currently logged in interactively. The pattern shows the attacker obtaining a TGT for a user who isn't actually present on that machine.
🔴 RED TEAM
Why Overpass-the-Hash: In environments that block NTLM (NTLMv1/v2 disabled, Kerberos enforced), standard PtH doesn't work. Overpass-the-Hash converts the stolen NTLM hash into a Kerberos ticket — bypassing the NTLM restriction. The result is a full Kerberos TGT that can be used for Pass-the-Ticket attacks across all Kerberos-enabled services.

SSH Tunnelling / Linux Lateral Movement
LATERAL MOVEMENT · REMOTE SERVICES · PROTOCOL TUNNELLING
▼
🔵 CORE BASELINE COMPOSITE
SSH Anomalous Source / Dynamic Port Forward Anchor
📎 INSERT RULE LINK · KQL · Sentinel + MDE 🔗 T1021.004 SSH 🔗 T1572 Protocol Tunnelling
🔵 BLUE TEAM
SSH is the SMB equivalent for Linux/Unix environments — the lateral movement substrate of choice. Minimum truth: SSH connections between internal hosts outside normal management patterns, SSH from a Windows host (unusual unless explicitly permitted), SSH with key authentication to multiple hosts in rapid succession (lateral movement sweep pattern), or SSH -D (dynamic port forward) or -L (local forward) flags creating tunnels.

In hybrid Windows/Linux environments, SSH becomes a pivoting substrate to reach Linux servers from an initially compromised Windows host.
🔴 RED TEAM
SSH dynamic tunnel (SOCKS proxy):
ssh -D 1080 -N user@linux-server → creates SOCKS5 proxy through the SSH session → all attacker traffic tunnelled through the compromised Linux host → appears as legitimate SSH management traffic.

SSH key theft: Steal ~/.ssh/id_rsa keys from compromised Linux hosts → use keys to authenticate to all other Linux servers those keys have access to → no password required, keys not rotated.
